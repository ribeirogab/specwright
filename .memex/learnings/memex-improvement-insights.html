memex improvement insights

Benchmarked against tlc-spec-driven (markdown sibling, deeper per-task rigor) and OpenSpec (compiled CLI, mechanical teeth). Pattern: memex wins on lifecycle governance, loses on two things both peers do better — closing the loop from acceptance criteria to verified implementation, and defending against the agent's own failure modes. Below: only the high-impact, concretely-actionable transfers.

Recommendations — ranked, high-impact only

1

Acceptance-criteria traceability IDs HIGH

from tlc-spec-driven   artifact: spec template · tasks template · memex-code-review · review-spec

Gap: AC are an unnumbered checklist; nothing links a criterion to the task that satisfies it or proves at review that every AC was verified.

Change: stable AC-1… IDs in the spec template; tasks reference the AC IDs they satisfy; code-review fails if any AC ID is unreferenced or unverified (coverage line X total, Y mapped, Z unmapped ⚠️).

2

AI-failure-mode test-integrity check HIGH

from tlc-spec-driven   artifact: AGENTS.md quality gate · memex-code-review

Gap: the quality gate has no guard against the agent's worst habit — deleting/skipping/weakening tests to make the gate go green.

Change: the touched area's test count must not drop and assertions must not be weakened/skipped without an in-spec justification; code-review flags a silent test deletion as a blocker. In autonomous mode the agent grades its own work — this is the highest-leverage trust mechanism, and memex lacks it entirely.

3

Spec-conformance "verify" dimension in code-review HIGH

from OpenSpec   artifact: memex-code-review SKILL

Gap: code-review checks the diff against project law, not against the spec's own acceptance criteria.

Change: before the verdict, walk the spec's AC (by ID from #1) + the plan's design decisions and report Completeness / Correctness / Coherence. "Passes lint + matches rules" ≠ "does what the spec promised."

4

Mechanical validator for spec/plan/tasks artifacts HIGH

from OpenSpec   artifact: skills/memex/scripts/ · review-spec

Gap: memex's own mechanical-enforcement-over-prose learning says runnable checks beat rules — yet spec artifacts are validated only by prose self-review.

Change: a small validator (repo already runs Python validators via uv run --with pyyaml) that checks required frontmatter, no surviving {{placeholder}}, AC present and free of banned vague verbs, and (with #1) every AC ID referenced by a task. Wire it into /memex:review-spec as a feedforward gate. Converts the most-rationalized-past failures into noisy ones — the Rule of Repair.

5

Mid-flight scope safety valve MEDIUM

from tlc-spec-driven   artifact: AGENTS.md Workflow decision block

Gap: the entry gate is binary; a "go direct" change that turns out large just keeps going ad-hoc.

Change: a direct change that, once underway, exceeds a small budget (e.g. >3 files or >5 non-trivial steps) must STOP and re-enter the spec flow. One prose rule, no new machinery.

6

Anti-fabrication knowledge-verification rule MEDIUM

from tlc-spec-driven   artifact: rules.md

Gap: rules.md has a Currency rule but no explicit anti-fabrication chain.

Change: verify factual/API claims against codebase/docs/Context7 before asserting; when unverifiable, flag uncertainty rather than inventing. memex already wires Context7 — this makes the discipline a stated rule.

Considered & parked — deliberately not recommended

Delta-spec merge into a living source of truth (OpenSpec) — conflicts with the constitution ("specs never get deleted; shipped specs remain as historical record"). memex's living layer is the learnings vault.
Externalized YAML workflow schemas / profiles (OpenSpec) — memex's flow is already editable markdown; a schema engine adds a build layer the "no build pipeline" rule resists.
Session HANDOFF.md checkpoint (TLC) — memex already prints a compact handoff prompt; redundant.
Brownfield 7-doc mapping (TLC) — heavy; conventions/ + /memex:recall partially cover it. Future, not now.
Graded complexity tiers as a full matrix (TLC) — the binary gate + #5 capture most of the benefit without the bookkeeping.
Filesystem-derived status engine (OpenSpec) — requires a CLI; frontmatter + handoff already give resumability at memex's scale.

How to apply

Treat #1–#4 as a coherent bundle: traceability IDs (#1) give #2/#3/#4 something concrete to check; together they convert memex's quality gate + code-review from "law compliance" into "law compliance + verified delivery of the spec." Implement as separate future specs (this benchmark is analysis-only). Start with #4 (mechanical spec validator) and #2 (test-integrity check) — both small, both turn a rationalized-past failure into a noisy one, both reinforce the differentiator memex already claims.
