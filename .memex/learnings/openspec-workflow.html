OpenSpec — Fission-AI spec-driven CLI

The structural opposite of memex: a compiled TypeScript CLI + external YAML+Markdown schemas, where thin agent skills shell out to the CLI for deterministic work. Model: "fluid not rigid, brownfield-first." State is derived from file existence; dependencies are enablers, not gates; the unit of work is a delta spec that merges into a living source of truth on archive.

Medium

Compiled CLI @fission-ai/openspec + Zod-validated schema.yaml + templates.

Core idea

Actions, not phases. Re-run openspec status to recover state anytime.

Memory

The filesystem. Delta specs merge into openspec/specs/ — a living spec base.

Enforcement

mechanical Zod schema + transactional merge — on the CLI path.

The delta-spec model

Instead of rewriting a whole spec, a change declares only ## ADDED / MODIFIED / REMOVED / RENAMED Requirements. Two changes can touch one capability without conflict; on archive the deltas merge into the living openspec/specs/ base — the spec set grows change-by-change and always reflects current behavior.

Format footguns: scenarios need exactly 4 hashtags (####) or they fail silently · MODIFIED with partial content silently loses detail · REMOVED on a brand-new spec is ignored.

Enforcement (mechanical, CLI-side)

  • Zod schema — required version, mandatory per-artifact template, unique ids, valid requires, DFS cycle detection.
  • Delta validation — section headers required, ≥1 scenario each, ADDED/MODIFIED need SHALL/MUST, duplicate/cross-section conflicts are hard errors.
  • Transactional merge — prepare+validate all, then write; abort with "No files were changed."
  • Soft gates — archive warns on incomplete tasks but does not block. Blocking is reserved for spec-corrupting input.
Asymmetry: a missing scenario on a main spec requirement is only a WARNING, but on a delta ADDED/MODIFIED it's a hard ERROR.

Strengths & limitations

Strengths

  • Genuinely brownfield — living, auditable spec base.
  • Resumable by design — filesystem-derived status.
  • Mechanical correctness (Zod + transactional merge, CLI path).
  • Portable across 25+ assistants; customizable schemas without forking.

Limitations

  • No human review gate / branch-PR cycle baked in.
  • Default /opsx:archive merge is agent-driven (mv), not the CLI engine.
  • No constitution / rules / learnings vault.
  • Quality gates not enforced; verify is advisory prose.
