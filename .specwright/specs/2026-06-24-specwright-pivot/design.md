# specwright — Design

## Purpose

Turn the skill into a **spec-driven-only** workflow installer and rename it to **specwright**. The product gives any repository one disciplined pipeline for every non-trivial change — `brainstorm → design → spec + tasks → implement → quality gate → PR → review-to-lgtm` — and nothing else. The entire knowledge-memory half (learnings and the vault that served it) is removed.

The name carries the product: **specwright** = the guardian of the spec — the thing that watches over every change so it honors its spec (quality gate, spec-conformance, review-to-`lgtm`). The short technical handle is **sw** (you type `sw`, you say `specwright`).

## Motivation

The skill currently does two jobs: a spec-driven workflow and an externalized project memory (the learnings vault, Obsidian wikilinks, maps-of-content, cross-linking). The two are independent, and the memory half carries most of the surface area and conceptual weight: four commands exist only to serve it (`recall`, `link`, `sweep`, `learn`), plus the Obsidian machinery, the `_index/` MOCs, and the reflection step.

Dropping the memory half leaves a sharper, smaller, more legible product: one job, done well. It also lets a cleaner architecture emerge — **the vault holds only living project state; all machinery ships inside the skill**. Scripts, spec templates, and the universal coding standard stop being scaffolded into every target repo and instead travel with the skill. What a target repo receives shrinks to `AGENTS.md` plus a two-folder vault.

This is treated as a brand-new skill. There is no migration path, no compatibility alias, and no reference to the old name anywhere in the product.

## Definitions

- **specwright** — the product/brand name. Appears in the vault directory (`.specwright/`), the marketplace, the README, and prose.
- **sw** — the technical handle. The plugin is named `sw` (Claude derives the command namespace from the plugin name), so commands are `/sw:spec`, `/sw:new-pr`, etc.; the scaffolder skill is invoked as `/sw`; companion skills live under `sw-<name>`.
- **vault** — the per-repo `.specwright/` directory. After the pivot it holds exactly two living things: `conventions/` (project-specific style) and `specs/` (specs and their history). Spec status lives in each spec's own frontmatter; there is no separate tracker.
- **machinery** — everything that is not living project state: validation/update scripts, spec templates, the universal coding standard. It ships inside the skill, not into the target repo.
- **the three copies** — every companion skill is kept in sync across `.agents/skills/sw-<name>/` (canonical), `plugins/sw/skills/<name>/` (Claude plugin), and `skills/sw/scaffold/skills/sw-<name>/` (what new installs receive).

## Non-Goals

- **No migration.** No converter, no alias, no backward compatibility, no upgrade path from the old skill.
- **No reference to the old name** anywhere in the product surface (skills, vault, docs, commands, prose).
- **No memory features.** Learnings, the knowledge vault, Obsidian wikilinks/backlinks, maps-of-content, and cross-linking are gone, not relocated.
- **No GitHub repo rename inside this change.** Renaming the remote repository (and thus the install URL) is the maintainer's out-of-band step; the code change only assumes the new name.
- **No new workflow steps.** The surviving flow keeps its shape; this pivot removes and renames, it does not add capabilities.

## What is removed

The memory half and its accessories: the learnings flow and the `recall`, `link`, `sweep`, `learn` commands; `rules.md`; `constitution.md`; `_index/`; `spec-driven-development.md` (AGENTS.md becomes self-contained); the `specs.md` tracker; all Obsidian (`.obsidian/*.json`, `[[ ]]` wikilink syntax, `related:` frontmatter, the `.gitignore` line, the Obsidian validation checks); note templates; the reflection/learnings step of the flow; and the "Vault — read from it, write to it" section of `AGENTS.md`.

## What moves into the skill

Following **vault = living state; machinery = ships in the skill**: the validation and update scripts; the spec templates (generated by `brainstorming`/`writing-plans`); and the universal coding standard — Unix philosophy, the meaningful-comments rule, and basic security — baked into the `code-review` skill as its own rubric. Project-specific standards remain in `conventions/`.

## What is kept

Companion skills `brainstorming`, `writing-plans`, `new-pr`, `code-review`, `review-spec`, `update`. Workflow law (commit/consent/git/PR conventions) lives in `AGENTS.md`. Specs are self-contained — no cross-references between them.

## Dogfood

This repository hosts the skill on itself. As a clean slate, the existing `.memex/` is deleted and a fresh `.specwright/` is born with this pivot's spec as its first record; still-true conventions are re-authored without any reference to the old name. Git history retains everything prior.
